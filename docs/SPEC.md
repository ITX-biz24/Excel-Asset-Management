# 個人資産管理システム v2 — 設計仕様書 (SPEC)

> 対象ファイル: `Excel-Asset-Management.xlsm`（現行 `Excel-Asset-Management.xlsx` を全面リファクタリング）
> 目的: 「一生使える個人資産管理システム」を、保守性・可読性・拡張性を最優先に構築する。
> 最終更新: 2026-07-09

---

## 0. 前提と方針

- 成果物は **`.xlsm`（マクロ有効ブック）**。
- VBA は **ソースコードを `vba/` にテキスト（`.bas`/`.cls`）で保存**し、Git で差分管理する。
- ブックは **ビルドスクリプトで再現生成**する（`build/`）。手作業でセルを触らず、設計＝コードで一元管理する。
  - `build/build_structure.py`（openpyxl）: シート・数式・入力規則・条件付き書式・グラフ・名前付き範囲を生成 → 中間 `.xlsx`。
  - `build/inject_vba.py`（Excel COM）: VBA モジュール注入・ボタン配置・テーマ微調整 → `.xlsm` 出力。
  - `build/build.py`: 上記を順に実行するオーケストレータ。
- 口座は **マスタ表で可変**。`現金 / 銀行 / QR` は初期値にすぎず、行を足すだけで口座を追加できる。
- 通貨は日本円（¥）。表示言語は日本語。

## 1. 現在の Excel の問題点

現行ファイルは単一シート `Sheet1` に全機能を詰め込んだ「1か月・31行固定」の家計簿である。動作はするが、拡張・保守の観点で以下の問題を抱える。

**構造・データモデル**
- 1シートに 2026年7月の1か月分しか持てない。翌月は手作業でシートを複製する前提で、履歴の通期集計ができない。
- 取引行が **7〜37 行の 31 行固定**。月末が31日でない月や取引が32件以上の月に対応できない。
- 「取引テーブル」がなく、`SUMIF`/`IF` を行数ぶんベタ書きしているため、行の増減で数式が壊れる。
- 口座 `現金 / 銀行 / QR` が数式・入力規則に**文字列直書き**。口座追加＝全数式の手修正が必要で、拡張不能。
- 取引の「種類（収入 / 支出 / 振替）」という概念がなく、出金元・入金先の埋まり方から暗黙的に判定しているだけ。振替と収支の区別が集計に反映されない。

**入力品質・エラー対策**
- 入力規則は出金元・入金先のドロップダウンのみ。金額0以下・残高不足・同一口座振替・カテゴリ/日付の未入力を防げない。
- カテゴリ・支払方法の概念がなく、支出分析ができない。
- 数式セルが保護されておらず、誤って上書きすると壊れる。

**機能不足**
- ダッシュボード、グラフ（資産割合・カテゴリ別・月別収支・資産推移）が無い。
- 固定収支・サブスク管理・年間集計・各種ランキング・資産スナップショットが無い。
- VBA が一切無く、翌月作成・固定反映・バックアップ・保護などの自動化が無い（README は「VBAで開発」と書くが実体は数式のみ）。

**保守性**
- レイアウトが大量のセル結合（200箇所超）に依存し、行追加・列追加でレイアウトが崩れる。
- ヘッダ表記が不統一（`収入`/`支出` が金額列の見出しになっているなど）で、意味を追いにくい。

## 2. 新しい設計案

### 2.1 設計原則

1. **単一の取引テーブルを唯一の真実（Single Source of Truth）**とする。すべての残高・集計・グラフはここから導出する。
2. **口座・カテゴリ・支払方法・種類はマスタ管理**し、名前付き範囲で全シートの入力規則・数式に供給する。文字列を直書きしない。
3. **残高は 2 本の SUMIFS で一般化**する（後述）。口座を増やしても数式は不変。
4. **入力セルと計算セルを色と保護で明確に分離**。計算セルはロックし、入力セルだけ編集可能。
5. **自動化は VBA、集計は数式**という役割分担。数式で完結することは VBA に持ち込まない（再計算で常に正しくなる）。
6. **ダークモダンなカード UI**。枠線・見出しを消し、KPI をタイル表示、色で意味を伝える。

### 2.2 残高計算の一般化（設計の核）

取引の種類ごとに口座列の使い方を統一する。

| 種類 | 出金元 | 入金先 |
|------|--------|--------|
| 収入 | （空欄） | 入金先のみ |
| 支出 | 出金元のみ | （空欄） |
| 振替 | 出金元 | 入金先（≠出金元） |

この規約により、任意の口座 `acc` の現在残高は **口座に依存しない 2 本の式**で表せる。

```
現在残高(acc) = 初期残高(acc)
              + SUMIFS(取引[金額], 取引[入金先], acc)   ' 入ってきた分（収入＋振替入金）
              - SUMIFS(取引[金額], 取引[出金元], acc)   ' 出ていった分（支出＋振替出金）
```

- 収入は入金先だけ、支出は出金元だけを埋めるので、この2式が自動的に全種類を正しく処理する。
- 振替は同額が入金先へ＋・出金元へ−となり、総資産は増減しない（内部移動）。
- **口座を追加しても式は 1 文字も変わらない**（マスタに行を足し、同じ式をコピーするだけ）。総資産＝口座残高の合計。

収入・支出の月次集計は種類で明示的に絞る（振替を除外するため）。

```
今月収入 = SUMIFS(取引[金額], 取引[種類], "収入", 取引[年月], 対象年月)
今月支出 = SUMIFS(取引[金額], 取引[種類], "支出", 取引[年月], 対象年月)
今月収支 = 今月収入 - 今月支出
貯蓄率   = IF(今月収入>0, 今月収支/今月収入, "")
```

## 3. シート構成

保守性のため役割ごとにシートを分割する。左から表示順。

| # | シート名 | 役割 | 主な内容 |
|---|----------|------|----------|
| 1 | **Dashboard** | 一覧・可視化 | KPIカード（総資産/口座別残高/今月収入・支出・収支/貯蓄率）、4グラフ、更新ボタン |
| 2 | **取引履歴** | 明細入力 | 取引テーブル（ListObject `T_取引`）。日付・種類・出金元・入金先・金額・カテゴリ・支払方法・メモ・年月 |
| 3 | **固定収支** | 定期収支 | テーブル `T_固定`。名称・種類・毎月/日付・金額・口座・カテゴリ・支払方法・メモ・有効。「今月へ反映」ボタン |
| 4 | **サブスク** | 定期課金 | テーブル `T_サブスク`。名称・開始日・支払日・金額・支払方法・口座・カテゴリ・ON/OFF。「今月へ反映」ボタン |
| 5 | **レポート** | 集計 | 月次サマリ・年間履歴・年間収入/支出/貯蓄・カテゴリランキング・支払方法ランキング |
| 6 | **スナップショット** | 資産推移 | 月末ごとの口座別残高・総資産（`T_スナップ`）。資産推移グラフの元データ |
| 7 | **マスタ** | 参照元 | 口座（`T_口座`：口座名・初期残高・表示順・現在残高）、カテゴリ（`T_カテゴリ`）、支払方法（`T_支払方法`）、種類（固定リスト） |
| 8 | **設定** | 環境設定 | 対象年月・バックアップ先・アプリ情報など。数式で使う定数を名前付き範囲で公開 |

補足:
- **初期残高は `マスタ!T_口座` に一元化**（設定に散らさない）。設定シートからは参照のみ。「口座はマスタで可変」を実現する唯一の置き場。
- `レポート` の集計用中間表（カテゴリ別・月別など、グラフの元データ）は同シート右側または最下部にまとめ、必要に応じて列を隠す。

### 名前付き範囲（抜粋）

| 名前 | 参照先 | 用途 |
|------|--------|------|
| `口座リスト` | `マスタ!T_口座[口座名]` | 出金元・入金先・各表の口座入力規則 |
| `カテゴリリスト` | `マスタ!T_カテゴリ[カテゴリ]` | カテゴリ入力規則 |
| `支払方法リスト` | `マスタ!T_支払方法[支払方法]` | 支払方法入力規則 |
| `種類リスト` | `マスタ!$?$?:$?$?`（収入/支出/振替） | 種類入力規則 |
| `対象年月` | `設定!$B$2` | ダッシュボード・集計の基準月（`yyyy-mm`） |

## 4. 数式設計

### 4.1 取引履歴（`T_取引`）

| 列 | 種別 | 内容 / 数式 |
|----|------|-------------|
| 日付 | 入力 | 必須。日付。 |
| 種類 | 入力 | 入力規則 `種類リスト`（収入/支出/振替）。 |
| 出金元 | 入力 | 入力規則 `口座リスト`。種類=収入なら空欄。 |
| 入金先 | 入力 | 入力規則 `口座リスト`。種類=支出なら空欄。 |
| 金額 | 入力 | 入力規則で `>0`。 |
| カテゴリ | 入力 | 入力規則 `カテゴリリスト`。必須。 |
| 支払方法 | 入力 | 入力規則 `支払方法リスト`。 |
| メモ | 入力 | 任意。 |
| 年月 | 計算 | `=IF([@日付]="","",TEXT([@日付],"yyyy-mm"))` 集計キー。 |

テーブル参照で行増減に強い。条件付き書式で 種類=収入→緑、支出→赤、振替→青の行帯を付ける。

### 4.2 マスタ（`T_口座`）

| 列 | 種別 | 数式 |
|----|------|------|
| 口座名 | 入力 | 例: 現金 / 銀行 / QR（行追加で拡張） |
| 初期残高 | 入力 | 開始時点の残高 |
| 表示順 | 入力 | ダッシュボード表示順 |
| 現在残高 | 計算 | `=[@初期残高]+SUMIFS(T_取引[金額],T_取引[入金先],[@口座名])-SUMIFS(T_取引[金額],T_取引[出金元],[@口座名])` |

### 4.3 Dashboard

```
総資産   = SUM(T_口座[現在残高])
口座別残高 = INDEX/MATCH または T_口座 参照（表示順どおり）
今月収入 = SUMIFS(T_取引[金額], T_取引[種類],"収入", T_取引[年月], 対象年月)
今月支出 = SUMIFS(T_取引[金額], T_取引[種類],"支出", T_取引[年月], 対象年月)
今月収支 = 今月収入 - 今月支出
貯蓄率   = IFERROR(今月収支/今月収入, "")   ' 収入0はブランク
```

### 4.4 レポート

- **月次サマリ**: 年月の一覧（設定で範囲指定 or 直近12/24か月を生成）ごとに
  `収入=SUMIFS(...,"収入",年月)`, `支出=SUMIFS(...,"支出",年月)`, `収支=収入-支出`, `貯蓄率=収支/収入`,
  `月末総資産`（スナップショット参照）。
- **年間履歴 / 年間収入・支出・貯蓄**: 年キー（`TEXT(日付,"yyyy")` 相当）で `SUMIFS`。
- **カテゴリランキング**: カテゴリごとに `SUMIFS(金額,種類,"支出",カテゴリ,cat[,年月])` を算出し、`LARGE`＋`INDEX/MATCH`（または並べ替え）で降順表示。対象月・年の切替可。
- **支払方法ランキング**: 同様に支払方法で集計。

### 4.5 スナップショット（`T_スナップ`）

| 年月 | 現金 | 銀行 | QR | …口座 | 総資産 |
- VBA が月末に現在残高を追記（後述）。資産推移グラフの元データ。過去改変を避けるため値で保存。

### 4.6 入力規則（Data Validation）まとめ

| 対象 | 規則 |
|------|------|
| 日付 | 日付・必須（空欄警告） |
| 種類 | リスト（収入/支出/振替） |
| 出金元 / 入金先 | リスト `口座リスト` |
| 金額 | 10進 `>0`（0以下禁止） |
| カテゴリ | リスト・必須 |
| 支払方法 | リスト |

> 「支出→入金先空欄」「収入→出金元空欄」「振替→出金元≠入金先・同一口座禁止」「残高不足の拒否」は
> 単純なセル入力規則では表現しきれないため、**VBA `Worksheet_Change` の入力チェックで担保**する（4.6 は一次防御、VBA は二次防御）。

## 5. VBA 設計

`vba/` にソースを置き、ビルド時に COM で注入。責務ごとにモジュール分割。命名は英語 camelCase / PascalCase、コメントは日本語。

| ファイル | 種別 | 役割 |
|----------|------|------|
| `modConfig.bas` | 標準 | 定数一元管理（シート名・テーブル名・名前付き範囲・色・バックアップ先） |
| `modUtils.bas` | 標準 | 汎用ヘルパ（テーブル取得・最終行・設定値取得・メッセージ・二重反映防止キー） |
| `modInputCheck.bas` | 標準 | 入力チェック本体（種類別の空欄整合・同一口座振替・金額>0・残高不足・必須未入力） |
| `modFixed.bas` | 標準 | `ReflectFixedItems`（固定収支を今月へ反映）、`ReflectSubscriptions`（サブスク反映）。二重反映防止 |
| `modSnapshot.bas` | 標準 | `SaveMonthEndSnapshot`（現在残高をスナップショットへ値で追記） |
| `modMonth.bas` | 標準 | `RolloverToNextMonth`（当月をスナップ保存→対象年月を翌月へ→固定/サブスク反映） |
| `modBackup.bas` | 標準 | `BackupWorkbook`（タイムスタンプ付きで別ファイル保存） |
| `modProtect.bas` | 標準 | `ProtectFormulas` / `UnprotectAll`（計算セルをロック、入力セルは編集可） |
| `modUpdate.bas` | 標準 | `RefreshAll`（再計算・集計/グラフ更新・入力規則再適用・保護再適用のワンクリック） |
| `ThisWorkbook.cls` | ドキュメント | `Workbook_Open`（初期化・対象年月既定値）、`Workbook_BeforeSave`（任意で自動バックアップ） |
| `Sheet_取引履歴.cls` | ドキュメント | `Worksheet_Change` → `modInputCheck` を呼び、NG は取消＆メッセージ。年月列の自動補完補助 |
| `frmTransaction`（COMで生成） | フォーム | 取引入力ユーザーフォーム（日付・種類・口座・金額・カテゴリ・支払方法・メモ）。検証後にテーブル追記 |

**入力チェック（modInputCheck）の判定ルール**
1. 日付・金額・カテゴリが未入力 → NG。
2. 金額 ≤ 0 → NG。
3. 種類=収入 かつ 出金元に値あり → NG（出金元は空に）。
4. 種類=支出 かつ 入金先に値あり → NG（入金先は空に）。
5. 種類=振替 かつ（出金元空 or 入金先空 or 出金元=入金先）→ NG。
6. 出金が発生する種類（支出・振替）で、その時点の出金元残高 < 金額 → 残高不足で NG（設定で許可も可）。

**ボタン割り当て**
- Dashboard: 「更新」→ `RefreshAll`、「バックアップ」→ `BackupWorkbook`、「翌月へ」→ `RolloverToNextMonth`、「取引を追加」→ `frmTransaction` 表示。
- 固定収支: 「今月へ反映」→ `ReflectFixedItems`。
- サブスク: 「今月へ反映」→ `ReflectSubscriptions`。

## 6. リスク

| # | リスク | 影響 | 対策 |
|---|--------|------|------|
| R1 | COM で VBA を注入するには「VBAプロジェクトオブジェクトモデルへのアクセスを信頼」が必要 | ビルド不可 | ビルド前に `AccessVBOM=1` を設定（実施済み）。手順を README/build に明記 |
| R2 | UserForm(.frm/.frx) のインポートはエンコード・バイナリで壊れやすい | フォーム破損 | `.frm` を配布せず、**ビルド時に COM で MSForm を生成しコントロールを動的追加**（`AddFromString` でコードのみ流し込み） |
| R3 | 日本語コードの文字化け（cp932 と UTF-8） | コンパイルエラー | ソースは UTF-8 保存、注入は `CodeModule.AddFromString`（COM が Unicode で受け渡し）を用いファイル取込のエンコード依存を回避 |
| R4 | マクロ有効化が必要（利用者がマクロ無効だと自動化が動かない） | 機能不全 | 数式で完結する集計は VBA 非依存に。マクロ無効でもダッシュボードは表示可能に設計。README に有効化手順 |
| R5 | 履歴増大で SUMIFS が重くなる | 動作低下 | テーブル境界参照（列全体でなく `T_取引[列]`）で範囲を限定。手動再計算は不要な設計 |
| R6 | 固定/サブスクの二重反映 | 二重計上 | 反映時にメモ等へ反映キー（`FIX:年月:名称`）を書き、既存があればスキップ |
| R7 | 残高不足チェックが行順・振替連鎖に依存 | 判定ずれ | チェックは「その口座の現在残高」基準で実施。厳密な時系列残高は設定でON/OFF可能に |
| R8 | セル結合多用でレイアウト崩壊（現行の失敗） | 保守困難 | KPIカード以外は結合を最小化。データ領域はテーブルで管理し結合しない |
| R9 | グラフが動的テーブルを参照しきれない | 表示崩れ | グラフ元データはテーブル or 固定範囲の中間集計表にし、`RefreshAll` で再構築 |
| R10 | Excel COM 依存でビルド環境が限定（Windows+Excel必須） | 再現性 | ビルド要件を明記。生成済み `.xlsm` はリポジトリに同梱し、Excel 単体でも利用可能に |

## 7. 改善案（v2 で反映 / 将来拡張）

**v2 で反映するもの**
- 単一取引テーブル化・口座マスタ化による無限月・無限口座対応。
- 収入/支出/振替の明示的な種類と、振替を除いた正しい収支・貯蓄率。
- カテゴリ/支払方法の導入と各種ランキング・グラフによる可視化。
- ダークモダン UI、入力セルと計算セルの色・保護分離。
- 固定収支・サブスクの自動反映、月末スナップショット、翌月ロールオーバー、バックアップ、ワンクリック更新。
- VBA ソースの Git 管理＋ビルド再現性。

**将来拡張（設計上の余地として確保）**
- 複数通貨・資産クラス（投資/ポイント/暗号資産）の口座タイプ拡張（`T_口座` に「種別」列を足すだけ）。
- 予算管理（カテゴリ別の月次予算 vs 実績）。
- CSV インポート（銀行/カード明細の取込 VBA）。
- 目標貯蓄額と進捗ゲージ。
- 年間 PDF レポート出力。

## 付録: ビルド手順（開発者向け）

```
# 前提: Windows + Excel + Python(openpyxl)。AccessVBOM=1 済み。
python build/build.py          # 構造生成(.xlsx) → VBA注入 → Excel-Asset-Management.xlsm 出力
```

- `vba/*.bas` `*.cls` を編集 → `build.py` 再実行で `.xlsm` を再生成（差分は Git で追跡）。
- コミット単位は `feat/dashboard` `feat/transactions` `feat/fixed-income` `feat/vba` `feat/report` の機能別。
